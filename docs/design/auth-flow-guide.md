# Azen — Auth & Onboarding Flow Guide

**Status**: Implemented
**Endpoints**: 11 total (5 auth, 4 org, 2 user)

---

## API Endpoints Summary

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/auth/otp/send` | None | Send OTP to phone |
| POST | `/auth/otp/verify` | None | Verify OTP, get auth_code + org list |
| POST | `/auth/token/issue` | None (auth_code) | Exchange auth_code for JWT |
| POST | `/auth/token/refresh` | None (refresh_token) | Rotate token pair |
| POST | `/auth/logout` | None (refresh_token) | Revoke refresh token |
| POST | `/orgs` | None (auth_code) | Create org + become transporter |
| GET | `/orgs/current` | JWT | Get current org details |
| GET | `/orgs/current/members` | JWT (transporter) | List org members |
| POST | `/orgs/current/members/invite` | JWT (transporter) | Invite fleet_owner or driver |
| GET | `/users/me` | JWT | Get own profile |
| PATCH | `/users/me` | JWT | Update own name |

## Flow 1: New User (first time, no org)

```
1. POST /auth/otp/send        { phone: "+919876543210" }
   → 200 { message: "OTP sent" }
   → Check console for OTP (dev mode)

2. POST /auth/otp/verify       { phone: "+919876543210", otp: "482910" }
   → 200 { auth_code: "uuid", user: {...}, organisations: [] }
   → Empty org list = new user

3. POST /orgs                   { phone: "+919876543210", authCode: "uuid",
                                  name: "Anil Logistics", slug: "anil-logistics" }
   → 201 { org: {...}, access_token: "eyJ...", refresh_token: "..." }
   → User is now logged in as transporter
```

**Important**: auth_code is single-use. After step 3, it's consumed. Do NOT also call `/token/issue` with the same auth_code.

---

## Flow 2: Returning User (has org)

```
1. POST /auth/otp/send         { phone: "+919876543210" }
   → 200

2. POST /auth/otp/verify        { phone: "+919876543210", otp: "482910" }
   → 200 { auth_code: "uuid", user: {...},
           organisations: [{ org_id: "uuid", name: "Anil Logistics", role: "transporter" }] }
   → Non-empty org list = existing user

3. POST /auth/token/issue       { phone: "+919876543210", authCode: "uuid", orgId: "uuid" }
   → 200 { access_token: "eyJ...", refresh_token: "...", expires_in: 900 }
```

---

## Flow 3: User with Multiple Orgs

```
1. otp/send → otp/verify
   → organisations: [
       { org_id: "org1", name: "Anil Logistics", role: "transporter" },
       { org_id: "org2", name: "Ravi Freight", role: "fleet_owner" }
     ]

2. Mobile app shows org picker

3. POST /auth/token/issue       { orgId: "org1", ... }  ← user picks Anil Logistics
   → JWT issued with role: "transporter" for org1
```

To switch orgs: logout → redo OTP flow → pick different org.

---

## Flow 4: Invite a Member

```
Prerequisite: Transporter is logged in with valid JWT

1. POST /orgs/current/members/invite   { phone: "+919876543210",
                                          name: "Aditya Sharma",
                                          role: "fleet_owner" }
   → 201 { member_id: "uuid", user_id: "uuid", role: "fleet_owner" }
   → If phone doesn't exist in system, user is auto-created
```

Invited user can now login via OTP and will see the org in their list.

---

## Flow 5: Token Refresh

```
Access token expired (after 15 min):

1. POST /auth/token/refresh     { refreshToken: "old_refresh_token" }
   → 200 { access_token: "new_eyJ...", refresh_token: "new_...", expires_in: 900 }
   → Old refresh token is revoked
   → Use new tokens going forward
```

---

## Flow 6: Logout

```
1. POST /auth/logout            { refreshToken: "current_refresh_token" }
   → 200 { message: "Logged Out" }
   → Refresh token revoked, can't be used again
```

## Auth_code Rules

- Generated by `otp/verify` after successful OTP verification
- **5-minute TTL** — expires after 5 minutes
- **Single-use** — consumed by either `POST /orgs` OR `POST /auth/token/issue`, not both
- BCrypt hashed in database, never stored in plain text

---

## JWT Claims

```json
{
  "sub": "user-uuid",
  "org_id": "org-uuid",
  "member_id": "org-member-uuid",
  "role": "transporter",
  "sub_role": "member",
  "exp": 1740000900,
  "iss": "Azen.Api",
  "aud": "Azen.Mobile"
}
```

- Access token: 15-minute expiry
- Refresh token: 30-day expiry, SHA256 hashed in DB

## Error Codes

| Code | Error | When |
|---|---|---|
| 200 | — | Success |
| 201 | — | Resource created |
| 400 | INVALID_ROLE | Invite with role other than fleet_owner/driver |
| 401 | INVALID_OTP | Wrong OTP or expired |
| 401 | INVALID_AUTH_CODE | Auth code expired, used, or wrong |
| 401 | INVALID_REFRESH_TOKEN | Refresh token expired, revoked, or wrong |
| 403 | NOT_A_MEMBER | User not a member of the specified org |
| 403 | FORBIDDEN | Non-transporter trying to invite/list members |
| 404 | USER_NOT_FOUND | User not in AuthDb |
| 404 | ORG_NOT_FOUND | Org not in AppDb |
| 409 | SLUG_TAKEN | Org slug already exists |
| 409 | ALREADY_MEMBER | User already in this org |

## Edge Cases

### OTP
- Max 5 wrong attempts per OTP → locked out, must request new OTP
- OTP expires after 10 minutes
- Multiple OTPs for same phone → only latest unused one is valid

### Auth Code
- Expires in 5 minutes — mobile app should call token/issue or /orgs immediately
- Single-use — cannot be reused after consumption

### Token Refresh
- Old refresh token is revoked on each refresh (rotation)
- If someone steals a refresh token and uses it, the legitimate user's next refresh will fail → signals compromise

### Org Creation
- Slug must be unique (URL-friendly identifier)
- Creator automatically becomes transporter with "member" sub-role
- ShipmentRefSequence is auto-created for the org

### Member Invite
- Only transporters can invite
- Valid roles: "fleet_owner", "driver"
- If phone not in AuthDb, user is auto-created (they can login later via OTP)
- Same user can't be invited to the same org twice

## Database Tables Used

### AuthDb
- `users` — identity (phone, name, email)
- `otp_requests` — OTP + auth_code storage
- `refresh_tokens` — hashed refresh tokens

### AppDb
- `organisations` — company entities
- `organisation_members` — user ↔ org ↔ role mapping
- `shipment_ref_sequences` — auto-increment counters per org
